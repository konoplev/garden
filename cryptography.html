Theory

Symmetric

Any cipher is actually replacement or re-ordering

Replacement - you have an algorithm that is applied to data to replace each symbol with another one. The simplest example is a replacement table. Each alphabet letter has a corresponding replacement.

Re-ordering - you have an algorithm that is applied to data to reorder letters in the text.

There are steam and block ciphers

Stream - there is an algorithm generating random numbers. You give it a seed (your key) and then the algorithm generates (using the seed) a random number for each received symbol.

An old example of the stream cipher - RC4.

New example - Salsa20

A stream cipher is good for a stream sending via a network. But it works badly for a file on disk. To read the last 10 bytes you need to decipher the whole file.

Block cipher - 128 or 256 bit. Takes a block of text and cipher it to the block with the same length. The size of the block is determined by an algorithm.

Examples - DES (old) и AES (new)

DES - the key is too short

triple DES to get bigger block and key

AES - key size is longer than DES

Хэши

Функция, которую надо повторить на каждом блоке, при этом результаты чейнятся как в CBC.

Главное свойство хэша - он одного размера и при минимальном изменении сообщения, хэш меняется сильно.

Используют главным образом для подписи.

Еще используют, чтобы из случайного числа получить еще более трудноподбираемое. Если генератор случайных числе плохой, то ключ будет плохим, лучше его предварительно пропустить через хэш.

Symmetric vs Assymetric encription

Symmetric

the same key is used to encrypt and decrypt messages. The key should be private.

Assymetric

two keys: public and private. The public is used to encrypt a message. After the message is encrypted nobody can read it. Even a person who just encrypted it. The message could be decrypted only by a person who has a private key.

Hash

A message digest (short length from any length message)

One way trip. It’s not possible to de-hash.

Hash can be used to check identity. Hash before sending is the same as after receiving.

Or for checking passwords. Instead of storing passwords, you add some salt to the password get hash from them, and store the hash. To check password identity you apply the same function, get hash and compare it with hash in the database.

Digital signature

  • Prove the message was not changed

  • Prove the source of the message (Authentication)

  • Make sure the signature isn’t fake

Sign with the private key. The message doesn’t need to be encrypted. Nobody else can sign thins (obviously)

You have some text. You create a hash of this text. Encrypt it with the private key. Send it with text.

The recipient has your public key. The public key is used to decrypt the signature and get the hash. The same hash function is applied to the text and the result is compared with decrypted hash. If this is the same hash, the message is sent from Alice.

Cryptograpthing nonce

Random number. The server memorizes it and sends it to the client, the client uses the number to mix it to reply. The server knows the random number, so it can check that this is the response to this particular request.

Ephemeral keys

Asymmetric keys are difficult to use, it takes a lot of CPU to decrypt. Usually, symmetric keys are used. Asymmetric keys are used to encrypt the symmetric key and send it via unprotected networks. The client receives an encrypted symmetric key, decrypt it using the public key, and uses a symmetric key for future communication. The symmetric key should be changed often. That’s why it’s called ephemeral.
